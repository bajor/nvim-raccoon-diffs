## Definition of exact parity

For the same parsed patch and effective options, the standalone extension's
semantic engine must produce exactly the same:

1. Context and change groups.
2. Change-block boundaries.
3. Deletion and addition row pairings.
4. Decision to compute or suppress an inline diff.
5. jsdiff change-object sequence.
6. Highlighted UTF-16 ranges on the deletion side.
7. Highlighted UTF-16 ranges on the addition side.
8. Changed-text coverage after conversion to Neovim byte columns.

Pixel-perfect browser rendering is out of scope because Neovim uses native
syntax highlighting and extmarks instead of Shiki, HAST, CSS, and the DOM.
Describe the result as semantically exact, not pixel-identical.

Exact semantic parity applies to the pure planner. Host integration parity
means every supported Raccoon route uses the same planner output without
changing host review semantics.

Never claim exactness based only on a green local suite. Exactness requires zero
mismatches against the pinned executable oracle.

## Required row pairing

Reproduce `iterateOverDiff.ts` exactly.

A contiguous change block containing `D` deletions and `A` additions has
`max(D, A)` split rows.

For zero-based row index `i`:

- pair `deletion[i]` with `addition[i]` when both exist
- a deletion with no addition has whole-line deletion highlighting only
- an addition with no deletion has whole-line addition highlighting only
- never realign rows by similarity or content
- never use `vim.diff` to change this pairing

Example: two deletions and three additions produce these rows:

1. deletion 1 paired with addition 1
2. deletion 2 paired with addition 2
3. unpaired addition 3

Example: three deletions and one addition produce these rows:

1. deletion 1 paired with addition 1
2. unpaired deletion 2
3. unpaired deletion 3

This positional behavior is required even when an inserted line appears at the
start or middle of a block.

## Required inline algorithm

For every paired deletion and addition row, mechanically reproduce
`computeLineDiffDecorations` from the pinned release.

Use this exact sequence:

1. Apply the exact `cleanLastNewline` behavior.
2. Measure line length using JavaScript UTF-16 code units, not UTF-8 bytes,
   Unicode code points, display cells, or grapheme clusters.
3. Skip inline decorations when either side contains more than 1000 UTF-16 code
   units.
4. Keep a line containing exactly 1000 UTF-16 code units eligible.
5. Run `diffWordsWithSpace` from `diff@9.0.0`.
6. Apply Pierre's exact `word-alt` `pushOrJoinSpan` behavior.
7. Preserve the `isLastItem` behavior exactly.
8. Treat `item.value.length === 1` as one JavaScript UTF-16 code unit. Do not
   reinterpret it as one Unicode character or one whitespace character.
9. Produce exactly the upstream deletion and addition decorations.
10. Only after upstream UTF-16 spans are complete, convert them to valid
    zero-based, end-exclusive Neovim UTF-8 byte columns.

Do not add a second `diffChars` pass. The pinned Diffshub default is `word-alt`,
not an automatic word and character hybrid.

The following example must follow Pierre even if another result appears more
precise:

```diff
-old_timeout = calculate_timeout(config)
+new_timeout = calculate_timeout(options)
```

Expected strong deletion spans:

- `old_timeout`
- `config`

Expected strong addition spans:

- `new_timeout`
- `options`

Do not refine `old_timeout` and `new_timeout` to only `old` and `new`.

Character mode may be ported only if an existing upstream option requires it.
It must not replace the Diffshub default.

## Rendering ownership

- host whole deleted row remains `RaccoonDelete`
